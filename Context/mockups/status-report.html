For leadership · the short version

Where we are

The core workflow is built and usable end-to-end on a real database. Reference data flows in from Airtable; the differentiating performance loop and the two-way write-back are the main pieces still ahead.

The problem we're solving

The Social, Ads and Content teams merged. Leadership can't see what's being produced, by whom, or how it performs — work is scattered across Jira and 4+ Airtable bases. The portal consolidates the whole lifecycle into one place, and answers the question no existing tool does:

“Who edited this asset — and how did it perform?” — answered in a single view. That performance loop is our edge; every other DAM leaves that gap open.

What the portal is

A role-aware workspace with three audiences:

  • Editors & designers — a personal queue: pull the next item, move it through production, attach assets.
  • Managers — a prioritization board: confirm the algorithmic queue in ~5 min/day, assign, approve.
  • Stakeholders & agencies — read-only status + performance, no paid seat required.

Plus a Vishen media → clip engine: long-form video in, AI-proposed short clips out, converted to tickets on approval.

Product requirements · the settled spec

What we agreed to build

Settled across three meetings (Titus 1:1, Blinkwork session, Hackathon, Jun 22–25). These decisions are fixed inputs, not suggestions — several deliberately reverse earlier approaches.

1Intake — a conditional chain, order matters

The form enforces taxonomy at submit so prioritization scores are never wrong from missing tags:

Event Type→ Asset Type (filtered)→ Team Lead + Editor (auto, read-only)→ Dimensions (required)→ Title · Brief · CTA · Positioning · Audience · Due date

Priority and Assignee are never on the form — they're handled by the backend.

2Two status axes, never merged

Priority Status prio_status — owned by managers, external-facing (New Request → To be reviewed → In Queue → Assigned…).

Ticket Status ticket_status — owned by editors/designers, internal (Backlog → To Do → In Progress → Review → Approved → Shipping…).

3Lifecycle — every transition is audited

Requested→ Prioritized→ Assigned→ In Production→ In Review→ Approved→ Published→ Performance Tracked

Each move writes a ticket_events row (actor, from, to, note). Ship gate: one real request completes this end-to-end, entirely in-tool.

4Three role surfaces · one mandated header

Hard requirement. The first five columns of every list view are identical: Title · Priority · Assigned · Ticket Status · Priority Status. Not a style choice.

5Airtable sync rules

DataDirectionTrigger
employees, dimensions, event/asset types, DNAAirtable → Postgreswebhook + nightly reconcile
tickets, assets, approvals, shootsPostgres ↔ Airtableon write, batched ≤10, 429 backoff

Reference data is read-only in the app (edit in Airtable). Always store airtable_id; reconcile is upsert-on-airtable_id.

6Architecture — the HYBRID decision decided 25 Jun

Build the workflow surfaces now in this standalone Next.js + Postgres repo (valid under either architecture), and migrate the nouns to Blinkwork brain nodes later. Under the target model, reference nouns (Asset, Person, Event, Channel, Metric, Insight) become brain nodes and Airtable becomes a connector; only workflow state (tickets, queue_rank, statuses, approvals) stays app-owned permanently. The workflow decisions, scoring algorithm and UI mockups stay valid in both.

Prioritization · how the queue is ordered

Score = urgency × complexity

Phase 1 is manual-assisted: the algorithm produces a default order; managers spend ~5 min/day confirming it and handling reassignments. Auto-assignment covers only the ~20–30% of unambiguous cases.

# urgency from due-date proximity + event tier urgency_score = 0.5·due_proximity + 0.5·event_tier # complexity from effort, variants, whether a shoot is needed complexity_score = 0.3·effort + 0.2·variants + 0.5·shoot_flag priority_score = urgency_score + lead_time_adjustment(complexity_score)

Tie-breaks: earlier due date → higher event tier → earliest created_at (FIFO, so nothing starves). When a manager drags a row, queue_rank overrides priority_score for display order — editors always pull from the manager-confirmed order. Weights are starting values, to be tuned after one real week. Implemented in lib/tickets/scoring.ts.

Still open: the event-tier ranking for the urgency score is unconfirmed. Owner: Moniek
